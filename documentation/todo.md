

# 1. новый режим fss_compare.py - сравнить один каталог и один fss файл

важная фича - для проверки на повреждение файлов.

просто делает сравнение одного каталога и одного fss локального файла ".index_hash.yaml".

фактически это работает так:
в памяти создается "fss объект" - слепок каталога.
и затем запускается обычное сравнение указанного файла ".index_hash.yaml" и этого "fss объекта".

Это похоже как если бы сравнивались два файла ".index_hash.yaml".
Но в этом случае один файл создается на лету и хранится в памяти.

# 1. новая фича - кол-во файлов в директории.

У файлов есть свойство "размер файла".

У деректорий пусть будет Свойство "кол-во файлов в директории".
Это будет удобно и возможно пригодится в некоторых кейсах.

Не всегда есть возможность записать такое свойство.
Например если мы создаем перечень файлов без рекурсии, то узнать кол-во файлов нет возможности.
В таком случае этот параметр создаваться не будет.

Также когда мы "собираем" множество файлов в один (merge),
то в этот момент желательно обновлять это свойство на основе данных которые мы сливаем.

# 1. новая программа - синхронизация времени и аттрибутов

программа из сохраненного файла пытается применить аттрибуты на каталог.

время:
  дата создания
  дата записи

атрибуты:
  Linux права доступа и владельца
  NTFS/FAS32 аттрибуты

программа не делает "умного поиска" - просто смотрит по указанному пути и применяет.

-----

этот режим имеет смысл только для Wnd, поскольку в Linux нельзя изменить атрибут "даты создания".

режим умного применения "даты создания" (не путать с "дата модификации"):
в этом режиме делается анализ:
если дата создания в файле "fss" меньше (более старая дата) чем дата на файле,
тогда применяется эта дата.
если дата создания в файле "fss" больше (более новая дата) чем дата на файле,
тогда дата или не применяется или наоборот, новая дата записывается в файл "fss".

фича - fss_to_ftp_list.py
==============

создает FTP сервер, где появляется папка, которая изображает список файлов.
"fss_to_ftp_list.py" при первом запуске создает FTP сервер (локальный) - "fss FTP сервер".
в "fss FTP сервере" появляется папка, которая имеет название идентичное названию fss-файла.

можно запустить "fss_to_ftp_list.py" второй раз, передать в аргументах второй fss-файл.
и тогда на "fss FTP сервере" появится в корне еще одна папка - для второго файла.

__эти файлы нельзя читать или писать__.
это только список.

зачем это нужно?
чтобы посмотреть список визуально, через программу типа Double Commander или другую подобную  (Total Commander).
чтобы сделать сравнение двух списков через FreeFileSync - там хорошо отображаются структуры каталогов и статистика изменений.

мне не нравится структура типа дерева.
у нее много проблем:
- отступы сжирают место. на глубоких уровнях пробелы могут занимать больше места чем информация.
- найдя запись в таком файле очень сложно понять путь по которому файл расположен, и невозможно этот путь скопировать.
- для записи файла нужно сформировать все дерево в памяти.
- для загрузки нужно загрузить весь файл, нельзя обрабатывать его по кусочкам.
- дерево дает красоту в текстовом редакторе - слево получается настоящее дерево.
  но проблема в том что такими продвинутыми редакторами никто не пользуется.
  и по факту эта красота не нужна даже мне.
  если я хочу посмотреть дерево - то можно сконвертировать в формат lst(DiskDir) и смотреть в файловой браузере что гораздо удобнее и функциональнее.
  плюс всегда можно написать конвертер в дерево, это легко.
  ну и я обнаружил что реальные файлы в реальных кэйсах просто не отображают дерево, потомучто файл весит 10 мегабайт, и парсер текстового редактора не делает разбор такого большого обьема строк.
- не возможно сделать поиск по пути.
- очень сложно редактировать файл - если нужно добавить содержимое директории вручную то нужно возиться с отступами.
Поэтому нужно переделать формат в вид:
словарь путей (не рекурсивный), где каждая запись имеет имя в формате path где указан относительный путь.
Это даст преимущества:
- в текстовом редакторе по прежнему будет видно стркутуру, просто ее нельзя будет сворачивать.
- можно искать по пути
- можно сохранять фрагменты, и выкидывать их из памяти, чтобы оптимизировать расход памяти.
- можно парсить по кусочкам - вытаскивать по одной записи, чтото делать с ней, и выкидывать ее из памяти, и брать следующую запись.
- человеку легко соединять файлы и редактировать их. потомучто не нужно возиться с отступами. можно просто закинуть контент в конец файла и быть увереным что он корректно присоединился к общей информации.
