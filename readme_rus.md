eng shot description
================

File sync with smart compare and resolve conflict.
Sync from multiple 'destination'.

```
Concept:
  sync folder content special file (in root), this file content:
    sync source ID.
    list of all destination and date of last sync time of all destination.
    md5 from all "special file (in folder)".
    ??? list of all files in all destination and 'modification date' of this files.

  all folder content "special file (in folder)", this file content:
    folder ID.
    md5 sum from all files of this folder.

  sync step:
    1. check present and load special file from root.
    2. compare 'source ID' and 'source ID' in destination.
    3. check destination to changes.
      3.1. recalc MD5 of all files in 'destination'.
      3.2. compare MD5 and MD5 from "special file (in folder)".
      3.3. if detect changes:
      3.3.1 then for this file make flag 'sync conflict'.
      3.1.2. special addon:
      3.1.3. if file not present in folder and in folder detect new file with some MD5 - make rename procedure.
      3.1.4. after this analiz - add all new file as new file.
      3.1.5. add to remove list all lost files.
    4. check source to changes.
    4.1. see 3.1...
    5. for all files
    5.1. if flag 'sync conflict' present only in source - copy to destination.
    5.2. if flag 'sync conflict' present both - warn about conflict or copy to destination as 'conflict copy'.
```

Термины
=============

Таргет - каталог/путь синхронизации (папка или даже целый диск).

DUID - disk uniqu ID. Уникальный индефикатор носителя информации на котором лежит таргет. Определяется по: метке диска, размеру диска, по серийному номеру (это не всегда возможно), названию устройства (флешки имеют разные названия чипа).

Алгоритм синхронизации
==========================

Алгоритм предусматривает что таргеты могут быть синхронизированны другой программой синхронизации.

Алгоритм рассчитан что таргетов много (больше двух).

Алгоритм рассчитан что таргеты могут очень долгое время отсутствовать и затем в устаревший таргет нужно залить актуальные данные.

Алгоритм предусматривает что таргеты могут быть на одном компе и на одном винчестере (для резервирования и прочего).

Алгоритм предусматривает что таргет может быть на флешке которая мигрирует между компьютерами.

Алгоритм анализирует доступные внутренние временные метки и может обнаруживать отмотку времени назад - что считается невозможным.
Алгоритм при старте тянет время из инета чтобы удостовериться что время на машине актуально.
Алгоритм может игнорировать до 24 часов несоответствия при условии кратности часов - на случай разных часовых поясов (и глюков связанных с этим).
Алгоритм может игнорировать 1 секунду разницы во времени для разных ФС - учитывая их оссобености.

Таргеты определяются по DUID носителя информации.

Все таргеты синхронизации (и связанные с ними DUID) записаны в специальном файле конфигурации.


Файл информации о синхронизации с разными источниками
====================================

В бета версиях:
файл только один - лежит в корне таргета.

при обновлении просто перезаписывает весь файл.

В продвинутых версиях:
Изменения в файле анализируются и записываются только кластеры которые изменились. В файле добавляются выравнивающие пробелы чтобы уменьшить кол-во смещений данных.

Новые файлы добавляются в конец файла (чтобы избежать смещения данных).

Удаленные файлы - взводится флаг удаления, но запись не стирается (чтобы избежать смещения данных). Такая запись может быть заменена на выравнивающие пробелы.

Переодически происходит перезапись всего файла - с сортировкой данных, обновлением выравнивающих пробелов, удалением устаревших записей.

Лежит во всех папках если там больше 30 файлов, иначе информация размещается в файле выше.



MD5 CRC
=====
считает у всех файлов MD5 CRC
записывает CRC в "NTFS-поток", а также для каждого каталога кладет в его корень файл ".crc_control.csv".
в котором каждая строка содержит: имя файла, дату изменения файла, размер файла, CRC файла, дату последнего сканирования CRC.
при повторном прогоне просто проверяет CRC у всех файлов и делает "обслуживающие" действия.

если у файлы перестали открываться то останавливается с сообщением об ошибке.
если файлы содержат нули - сообщение об ошибке.

Проверка состояния флешки
=====================

Проверка состояния флешки, ее наличие и ошибки монтирования.
Отсутствия ее подмены: метке диска, размеру, по серийному номеру (это не всегда возможно).

`psutil.disk_usage(findTarget()).free == 0`


Автоматическая синхронизация при втыкании флешки
==================

И вообще при любом обнаружении таргета.

Есть два режима:

1. В памяти висит простой сервис обнаружения (а не вся программа). И он запускает программу.

2. Программа запускается кемто другим - например USB Safely Remove умеет запускать прогу при втыкании флешки.


Режим синхронизации структуры
===================================

Особый режим.
Считает MD5 для двух таргетов.
Затем анализирует различия.
Затем подготавливает список переименований/перемещений на втором таргете чтобы после этого второй таргет стал более похожим на первый таргет.
Причем чтобы список генерился также как bash команда.
А также список того чего не хватает во втором таргете.
Зачем нужно?
У меня есть два таргета, которые весят _очень_ много гигабайт/терабайт.
Первый таргет - рабочий.
Второй таргет - резервная копия.
До второго таргета очень слабый канал связи.
Хочется привести второй таргет к состоянию максимально близкому к первому таргету.
Чтобы потом тупые программы синхронизации сделали свою работу без лишних прокачек.
